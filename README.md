# Менеджер задач

React + Vite приложение для локального ведения задач. Поддерживает приоритеты, описание, связи между задачами, ручной hold, автоматический hold по зависимостям, закрытие задач и установку как PWA.

Приложение рассчитано на публикацию как статический сайт, например через GitHub Pages. По умолчанию каждый пользователь открывает одну и ту же ссылку, но его задачи хранятся локально в его браузере в `localStorage`.

Дополнительно приложение поддерживает онлайн-режим через Supabase. Локальный режим остается режимом по умолчанию. Онлайн-режим включается пользователем вручную и хранит задачи отдельно для каждого вошедшего пользователя.

## Стек

- React 19
- Vite 8
- ESLint 10
- vite-plugin-pwa
- GitHub Pages
- Supabase JS

## Команды

```bash
npm install
npm run dev
npm run build
npm run preview
npm run lint
```

`npm run dev` запускает сервер разработки. `npm run build` собирает production-версию и генерирует PWA manifest/service worker. `npm run preview` показывает собранную версию локально. `npm run lint` проверяет код.

## Онлайн-режим Supabase

Локальный режим работает без Supabase. Чтобы включить онлайн-режим:

1. Создайте проект в Supabase.
2. Откройте SQL Editor и выполните `supabase.schema.sql`.
3. Скопируйте `.env.example` в `.env.local`.
4. Заполните:

```bash
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=your-public-anon-key
```

5. Перезапустите dev-сервер или пересоберите приложение.

Для GitHub Pages добавьте эти же значения в репозитории: `Settings -> Secrets and variables -> Actions -> New repository secret`. Workflow передаст `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY` в `npm run build`.

В приложении появится переключатель `Локально / Онлайн`. В онлайн-режиме пользователь входит по email magic link. Задачи изолированы через `user_id` и Supabase Row Level Security: один пользователь не видит задачи другого.

Кнопка `Перенести локальные` копирует текущие локальные задачи в онлайн-хранилище. Локальные данные при этом не удаляются.

## Публикация на GitHub Pages

1. Создайте репозиторий на GitHub.
2. Инициализируйте git локально, если он еще не создан:

```bash
git init
git branch -M main
git add .
git commit -m "Initial PWA release"
git remote add origin https://github.com/<user>/<repo>.git
git push -u origin main
```

3. В GitHub откройте `Settings -> Pages`.
4. В `Build and deployment` выберите `Source: GitHub Actions`.
5. После push workflow `Deploy Pages` соберет приложение и опубликует содержимое `dist`.

Vite настроен с относительным `base: './'`, поэтому сборка работает как в корне домена, так и на GitHub Pages в подпапке репозитория.

## Установка как PWA

После публикации откройте GitHub Pages URL в поддерживаемом браузере. В адресной строке или меню браузера появится действие установки приложения. Установленная PWA запускается как отдельное окно, но данные по-прежнему остаются локальными для конкретного браузера и пользователя.

Если браузер показывает старое поведение после обновления, откройте DevTools -> Application и проверьте Service Workers и Storage. В обычном сценарии новый service worker активируется быстрее благодаря `skipWaiting` и `clientsClaim`.

## Работа с задачами

Новая задача получает средний приоритет. Значок приоритета слева открывает меню выбора: критический, очень высокий, высокий, средний или низкий. Закрытие и повторное открытие выполняется только через чекбокс.

Двойной клик по активной задаче или задаче на hold открывает детали: заголовок, приоритет, описание и раздел «Связи». Закрытые задачи нельзя редактировать, менять им приоритет или ставить на hold, пока они снова не открыты.

## Hold и связи

Правая кнопка мыши по основной области активной задачи ставит ее на hold или увеличивает hold на 15 минут. Удержание правой кнопки повторяет действие с ускорением. Левая кнопка по желтому таймеру открывает выбор точной даты и времени окончания hold, правая кнопка уменьшает hold на 15 минут вплоть до обнуления.

В деталях задачи связь «зависит от» указывает задачу-причину. Пока задача-причина не закрыта, текущая задача автоматически находится в hold-списке. Со стороны задачи-причины эта же связь отображается как «блокирует».

Если задача-блокер держит цепочку задач с более высоким приоритетом, в списке она визуально получает максимальный приоритет из всей открытой цепочки и сортируется по нему.

## Хранение данных

В локальном режиме задачи хранятся под ключом `task-manager.tasks` в версионированном формате `{ version, tasks }`. Старый формат в виде массива мигрирует автоматически при запуске. При чтении данные нормализуются: некорректные поля заменяются значениями по умолчанию, ссылки на удаленные задачи убираются, просроченные таймеры hold очищаются.

В онлайн-режиме задачи хранятся в таблице Supabase `tasks` для текущего `user_id`. Данные не входят в сборку и не передаются другим пользователям.

Удаление онлайн-задач выполняется мягко: задача скрывается в приложении, но строка остается в Supabase с `deleted = true` и `deleted_at`. Это нужно для будущих возможностей просмотра и управления удаленными задачами.

## Структура

```text
src/
  App.jsx       # логика задач, приоритетов, связей, hold и деталей
  App.css       # стили интерфейса
  main.jsx      # точка входа React
  index.css     # глобальные стили
public/         # favicon и PWA-иконки
dist/           # результат сборки, не хранится в git
.github/        # GitHub Actions workflow для публикации Pages
supabase.schema.sql # таблица и RLS-политики для онлайн-режима
```
